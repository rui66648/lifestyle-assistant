# AI 成长闭环 - 功能规格说明书

> 从「被动记录」到「主动成长」：基于数据的智能建议 → 用户确认 → 自动创建习惯

---

## 一、功能概述

### 1.1 核心理念

传统打卡APP是**工具**，用户记录什么就显示什么。AI成长闭环让APP升级为**个人成长教练**：

```
原始数据 → AI分析 → 洞察建议 → 用户确认 → 自动执行 → 效果验证
    ↑                                            ↓
    └──────────────── 持续迭代 ────────────────┘
```

### 1.2 为什么这是核心壁垒

| 维度 | 传统打卡APP | AI成长闭环 |
|------|------------|-----------|
| 用户角色 | 记录者 | 成长者 |
| APP角色 | 工具 | 教练 |
| 价值来源 | 用户自律 | 数据+AI驱动 |
| 留存逻辑 | 习惯惯性 | 持续惊喜 |
| 付费意愿 | 低（工具属性） | 高（成长伙伴） |

## 二、优秀案例参考

### 2.1 Habitify - 数据驱动的习惯教练

**亮点：**
- 周报自动生成，用图表展示完成率趋势
- "智能建议"基于历史数据推荐调整时间
- 连续失败3次触发"习惯诊断"

**可借鉴：**
- 周报机制：每周一自动推送，培养用户查看习惯
- 失败触发：不是等用户主动找，而是问题出现时主动介入

### 2.2 Streaks - 极简但精准

**亮点：**
- 只做12个习惯的上限，强迫用户聚焦
- 连击数可视化，用游戏化驱动坚持
- 小组件做得极好，降低打卡门槛

**可借鉴：**
- 建议数量控制：每周不超过2条，避免"建议疲劳"
- 可视化呈现：用进度环、热力图让成长看得见

### 2.3 Notion AI - 确认式交互

**亮点：**
- AI生成内容后，用户可以"保留/替换/取消"
- 编辑功能：不是只能接受或拒绝，可以调整参数
- 透明化：说明"为什么给这个建议"

**可借鉴：**
- 确认卡片三态：采纳 / 编辑后采纳 / 暂不需要
- 洞察说明：用一句话解释推荐理由
- 可编辑性：用户可以修改习惯名称、时间等参数

### 2.4 小日常 - 东方美学与游戏化

**亮点：**
- 成就系统设计精美，收集感强
- 习惯树/花园等成长可视化
- 温柔的提醒语气，不说教

**可借鉴：**
- 成就徽章与新习惯联动：添加第N个习惯解锁对应成就
- 语气温和：用"我们发现..."代替"你应该..."
- 仪式感：新习惯创建有欢迎动画

## 四、核心功能详细设计

### 4.1 多周期分析体系

#### 4.1.1 周报（Weekly Review）

**触发时机**：每周一 09:00 自动生成

**分析维度：**
| 维度 | 计算方式 | 洞察示例 |
|------|---------|---------|
| 完成率波动 | 本周vs上周对比 | "本周运动完成率下降40%，主要集中在周三周五" |
| 失败时间分布 | 按星期统计失败率 | "你在周三最容易放弃，试试把难度调低？" |
| 习惯关联性 | 双习惯完成率相关性 | "晚睡的日子，第二天早起打卡率下降60%" |
| 最佳表现 | 完成率最高的习惯 | "喝水习惯已连续21天，太棒了！" |

**建议数量**：1-2条（不超过2条）

#### 4.1.2 月报（Monthly Review）

**触发时机**：每月1日 09:00 自动生成

**分析维度：**
| 维度 | 计算方式 | 洞察示例 |
|------|---------|---------|
| 习惯稳定性 | 连续>21天标记为"稳定" | "晨起喝水已形成稳定习惯，可挑战更高目标" |
| 衰退预警 | 近2周完成率降幅>30% | "阅读习惯在第3周后下降60%，需要调整吗？" |
| 类别均衡 | 各分类完成率对比 | "运动类完成85%，但心灵类只有30%" |
| 里程碑 | 累计天数/连击数 | "你已累计打卡100天，超越了80%的用户" |

**建议数量**：2-3条

#### 4.1.3 年报（Yearly Review）

**触发时机**：每年1月1日 09:00 自动生成

**分析维度：**
| 维度 | 计算方式 | 洞察示例 |
|------|---------|---------|
| 年度关键词 | 基于习惯标签聚类 | "2025你的关键词：早睡早起、碎片阅读" |
| 季节性变化 | 按季度统计完成率 | "你在春季行动力最强，冬季容易懈怠" |
| 成长轨迹 | 月度完成率曲线 | "从3月开始稳步上升，下半年保持在85%+" |
| 价值观对齐 | 习惯类别占比vs目标 | "运动占比40%，但你说过想多读书..." |

**建议数量**：3-5条（战略方向型）

#### 4.1.4 微分析（实时触发）

**触发条件：**
- 某习惯连续失败 ≥ 3次
- 睡眠数据连续3天异常
- 完成率周环比下降 > 20%

**特点：**
- 轻量化，只有1条建议
- 即时性，问题出现24小时内推送
- 聚焦具体问题，而非全局分析

### 4.4 习惯自动创建

#### 4.4.1 创建流程

```
1. 用户点击"采纳并创建"
   ↓
2. 校验：检查是否已存在同名习惯
   ↓ （已存在 → 提示"已有此习惯"）
   ↓ （不存在 → 继续）
3. 生成完整习惯配置
   - 从建议中提取基础信息
   - 补全默认值（repeat、tip、category等）
   - 关联AI洞察作为习惯备注
   ↓
4. 调用 HabitModule.addHabit() 创建
   ↓
5. 历史数据回填（可选）
   - 如果是调整类建议，迁移原习惯历史数据
   - 如果有对应数据源（如步数、睡眠），自动补全过去7天
   ↓
6. 触发成就检查 + 更新等级
   ↓
7. 显示成功反馈 + "查看详情"入口
```

#### 4.4.2 历史数据回填策略

| 习惯类型 | 回填数据 | 回填天数 |
|---------|---------|---------|
| 步数/运动 | 从运动记录导入 | 7天 |
| 睡眠相关 | 从睡眠记录计算 | 7天 |
| 饮水 | 从饮水记录导入 | 7天 |
| 其他 | 不回填（从零开始） | 0天 |

> 注意：回填仅用于展示趋势，不计入成就和等级（避免"作弊"感）

### 4.5 反馈闭环

#### 4.5.1 反馈数据收集

| 事件 | 记录内容 | 用途 |
|------|---------|------|
| suggestion_shown | 建议ID、展示时间、周期 | 统计展示量 |
| suggestion_accepted | 建议ID、采纳时间、是否编辑 | 计算采纳率 |
| suggestion_rejected | 建议ID、拒绝时间、拒绝原因 | 优化推荐算法 |
| habit_7day_retention | 创建7天后的完成率 | 评估建议质量 |
| habit_30day_stability | 创建30天后是否稳定 | 长期效果评估 |

#### 4.5.2 自适应优化

```
采纳率 < 30% → 建议过于激进 → 降低难度/数量
采纳率 30-60% → 正常区间 → 保持当前策略
采纳率 > 80% → 建议过于保守 → 适当增加挑战性
```

#### 4.5.3 拒绝原因收集（可选）

用户点击"暂不需要"时，可选选择原因：
- 不感兴趣
- 太难了
- 没时间
- 已经在做了
- 其他（文本输入）

## 六、技术实现要点

### 6.1 数据最小化原则

传给AI的是**聚合指标**，不是原始数据：

```javascript
// ✅ 这样传（安全）
const aiInput = {
  weeklyCompletionRate: 0.72,        // 百分比
  exerciseDays: 3,                    // 天数
  avgSleepHours: 6.5,                 // 平均数
  trend: 'up'                         // 趋势
};

// ❌ 不要这样传（隐私风险）
const aiInput = {
  records: [                          // 逐条原始记录
    { date: '2026-07-14', sleep: '23:45', exercise: true },
    { date: '2026-07-15', sleep: '00:30', exercise: false },
    // ... 30条原始数据
  ]
};
```

### 6.2 降级策略

AI不可用时的兜底方案：

1. **本地规则引擎**：基于预设规则生成建议（无需网络）
2. **模板推荐**：从习惯库中按体质/季节推荐（现有recommendation.js已实现）
3. **友好提示**：显示"AI暂时休息中，为你准备了精选建议"

### 6.3 性能优化

| 优化点 | 方案 | 效果 |
|--------|------|------|
| 分析缓存 | 相同数据组合缓存24小时 | 减少90% API调用 |
| 懒加载 | 只在用户点击时才调用AI | 零无效调用 |
| 节流 | 同一用户每7天最多1次周报 | 控制成本 |
| 本地计算 | 数据聚合在本地完成 | 减少传输量 |

## 八、风险与应对

| 风险 | 概率 | 影响 | 应对方案 |
|------|------|------|---------|
| AI建议不准确 | 中 | 用户信任下降 | 降低置信度显示 + 允许用户反馈纠错 |
| 建议过多导致疲劳 | 高 | 用户忽略所有建议 | 严格控制数量（周报≤2条） |
| 隐私担忧 | 中 | 用户拒绝使用 | 数据最小化 + 本地模式 + 明确授权 |
| API成本过高 | 中 | 运营成本上升 | 缓存 + 节流 + 本地规则兜底 |
| 创建习惯后放弃 | 高 | 功能价值降低 | 历史回填 + 7天跟进提醒 + 调整机制 |
